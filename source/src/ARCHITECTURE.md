# Архитектура проекта USBBackupApp

## 1) Назначение

`USBBackupApp` — настольное Windows-приложение (Python + CustomTkinter) для резервного копирования между USB-носителем и выбранной папкой бэкапа.

Главная цель архитектуры: отделить:

- UI-слой (главное окно, кнопки, состояние интерфейса),
- файловую логику (сканирование, сравнение, копирование),
- платформенные интеграции (поиск дисков, автозапуск в Windows).

---

## 2) Структура модулей

### Точка входа

- `main.py`
  - тонкая точка входа: аргументы CLI, single-instance, UAC, запуск `CTk`.

### Ядро логики

- `core/backup_logic.py`
  - `BackupLogic.should_skip_dir()` — фильтрация системных папок;
  - `BackupLogic.scan_files_progress()` — рекурсивное сканирование каталога;
  - `BackupLogic.file_hash()` — SHA256 хеш файла;
  - `BackupLogic.copy_file_streaming()` — копирование файла по чанкам с прогрессом;
  - `BackupLogic.analyze_differences()` — анализ отличий между источником и бэкапом.
- `core/app_paths.py`
  - единая точка для путей проекта, ресурсов и состояния приложения;
  - чтение версии из файла `VERSION`.

### UI-компоненты

- `ui/main_window.py`
  - главный класс `BackupApp` — единственное окно приложения (Single-window UI);
  - оркестрирует все пользовательские сценарии: сканирование, копирование, автозапуск;
  - прогресс сканирования и копирования отображается в нижней панели главного окна;
  - отмена активной операции — через кнопку «Отмена» в той же панели;
  - блокирует левую панель управления во время выполнения операций.
- `ui/progress_presenter.py`
  - UI-агностичный форматтер ETA, размера (Б/КБ/МБ/ГБ) и скорости передачи.
- `ui/__init__.py`
  - пакет UI-компонентов.

### Интеграция с Windows

- `device_detector.py`
  - получение списка доступных источников (`USB` и `DISK`) через WinAPI;
  - формирование `device_id`;
  - получение `PNPDeviceID` через PowerShell для настройки автозапуска.

- `services/autolaunch_service.py`
  - создание/удаление задачи в Windows Task Scheduler (`schtasks`);
  - активация журнала `Microsoft-Windows-DriverFrameworks-UserMode/Operational`;
  - формирование команды запуска для dev- и frozen-режима.
- `services/transfer_service.py`
  - цикл копирования файлов с обработкой отмены и `PermissionError`;
  - коллбеки прогресса/успеха/ошибки без привязки к UI-виджетам.
- `services/scan_service.py`
  - сканирование source/backup и анализ отличий через `BackupLogic`;
  - возвращает готовый словарь `analysis` и метаданные для UI.

### Устаревшие модули (не участвуют в текущем основном потоке)

- `legacy/backup_engine.py`
- `legacy/index_db.py`
- `legacy/restore.py`
- `legacy/logger_utils.py`
- `legacy/__init__.py`

Эти файлы сохранены для референса и не используются главным сценарием `main.py`.

## 3) Поток выполнения (высокий уровень)

1. Запуск приложения:
   - `main()` в `main.py` обрабатывает аргументы (`--autostart-device-id` и др.),
   - проверяет single-instance для автозапуска,
   - при необходимости запрашивает повышение прав (UAC),
   - создаёт `CTk` и `BackupApp`.

2. Инициализация UI:
   - `BackupApp.__init__()` строит главное окно,
   - подгружает последнее состояние (папка бэкапа из `app_state.json`),
   - обновляет список устройств.

3. Сканирование:
   - пользователь нажимает «Сканировать папку»,
   - `BackupApp.analyze_backup_mode()` запускает фоновый поток `scan_worker`,
   - сканирование каталогов через `ScanService.analyze()` → `BackupLogic.scan_files_progress()`,
   - анализ отличий через `BackupLogic.analyze_differences()`,
   - результат записывается в `self.analysis`.

4. Копирование:
   - пользователь выбирает действие (в бэкап / на флешку),
   - `BackupApp._run_copy_job()` берёт операции из `self.analysis`,
   - `TransferService.run_copy_loop()` копирует файлы через `BackupLogic.copy_file_streaming()`,
   - прогресс (процент, ETA, скорость) отображается в нижней панели главного окна,
   - поддерживается отмена через `threading.Event`.

5. Завершение:
   - обновляется статус-строка и журнал в правой панели,
   - нижняя панель прогресса скрывается,
   - интерфейс разблокируется и возвращается в рабочее состояние.

---

## 4) Модель данных операций

`analysis` (в `BackupApp`) содержит:

- `mode`: `full` | `sync` | `none`;
- `ops_to_backup`: список операций `(src, dst, size, rel)`;
- `ops_to_usb`: список операций `(src, dst, size, rel)`;
- `bytes_to_backup`, `bytes_to_usb`;
- служебные поля (`extra_*`, `matched`, `common_paths` и т.д.).

Это центральный контракт между этапом анализа и этапом копирования.

---

## 5) Параллелизм и отмена

- Долгие операции (сканирование/копирование) выполняются в фоновых потоках (`threading.Thread`).
- Для кооперативной остановки используется `threading.Event`:
  - `scan_cancel_event`,
  - `transfer_cancel_event`.
- Обновление UI выполняется через `root.after(...)`, чтобы не обращаться к Tk из рабочего потока напрямую.

---

## 6) Single-window UI и жизненный цикл прогресса

Всплывающих окон нет. Вся индикация сканирования и копирования находится в нижней панели главного окна (`progress_frame`):

- панель показывается (`pack`) при начале операции и скрывается (`pack_forget`) по завершении;
- во время выполнения левая панель блокируется (`state="disabled"`);
- кнопка «Отмена» активна только во время выполнения операции;
- текущая операция отслеживается через `self.active_operation` (`"scan"` | `"copy"` | `None`).

---

## 7) Логирование

- Пользовательский журнал с цветовой индикацией (успех/ошибка/система) выводится в правой панели главного окна (`log_box`).
- Журнал поддерживает автопрокрутку: при прокрутке пользователем вверх автоследование отключается, при возврате вниз — включается обратно.
- Отладочная runtime-инструментация (NDJSON-логи) используется только в debug-сессиях и удаляется из кода перед релизом.

## 8) Где менять код под типовые задачи

- Изменить внешний вид/поведение прогресса сканирования или копирования:
  - `ui/main_window.py` → методы `_update_scan_ui`, `_update_progress_ui`

- Изменить форматирование ETA, скорости, размера:
  - `ui/progress_presenter.py`

- Изменить алгоритм сравнения/копирования:
  - `core/backup_logic.py`

- Изменить общий пользовательский сценарий и кнопки:
  - `ui/main_window.py` → `BackupApp`

- Изменить правила определения устройств:
  - `device_detector.py`

- Изменить логику автозапуска через Task Scheduler:
  - `services/autolaunch_service.py`

- Изменить цикл копирования и обработку ошибок передачи:
  - `services/transfer_service.py`

## 9) Ограничения и известные особенности

- Приложение ориентировано на Windows (WinAPI, PowerShell, Task Scheduler).
- При работе с синхронизируемыми папками (например OneDrive) возможны редкие `PermissionError` на отдельных файлах; такие файлы пропускаются с предупреждением и отображаются в итоговом диалоге.
- Алгоритм сравнения определяет режим (`full`/`sync`) на основе коэффициента схожести: если менее 50% файлов совпадают — выбирается полный бэкап.
- При совпадении размеров файлов, но расхождении mtime, для точной проверки вычисляется SHA256-хеш.

## 10) План дальнейшего развития

Следующий крупный архитектурный шаг — снижение размера `ui/main_window.py` за счёт выноса отдельных зон ответственности:

- `ui/actions.py` — кнопочные сценарии (scan, copy, autolaunch),
- `ui/state.py` — состояние и StringVar-привязки,
- `services/app_state.py` — чтение/запись пользовательского состояния и миграции пути.

Это упростит сопровождение и ускорит адаптацию новых разработчиков.
